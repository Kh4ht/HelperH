📘 SAVE SYSTEM DOCUMENTATION (Unity + JSON + Migration + Autosave)
Overview

This save system handles:

JSON saving/loading

Optional AES encryption

Autosaving

Versioning

Backups

Data migrations

Serializable dictionaries

It is designed to be safe, modular, and future-proof.

🧩 Core Components
1. SaveData.cs

This is the data structure that defines everything you save.

Example:

[Serializable]
public class SaveData
{
    public int version = SaveManager.CURRENT_SAVE_VERSION;

    public string selectedHeroID;
    public HeroData heroData;

    public int coins;
    public int gems;
    public List<string> unlockedTowers = new();
    public SerializableDictionary<string, int> towerLevels = new();

    // Add new fields here
}

✔ Notes

version is extremely important: it's used to decide whether migration is needed.

Any field added later must be handled with a migration for old saves.

2. SaveManager.cs

Handles:

Reading & writing JSON

Encryption/decryption

Atomic saving (temp file + backup)

Running migrations

Keeping a global saveData

Loading Flow
Load() called →
    If save.json does not exist → create new empty save → call Save()
    Else →
         1. Read save.json
         2. If encrypted → decrypt it
         3. JsonUtility.FromJson()
         4. If version < CURRENT_SAVE_VERSION → run migration
         5. Save to backup
         6. Assign to SaveManager.saveData

Saving Flow
Save() called →
    1. Serialize saveData → JSON
    2. If encryption enabled → encrypt JSON
    3. Write to temp file
    4. Move old save to backup
    5. Move temp to save.json

Why atomic saving matters?

Prevents corruption if:

Game crashes while saving

Player closes game

Power loss

3. SaveAutoSaver.cs

A MonoBehaviour that:

Calls Load() on startup

Saves periodically

Saves on quit/pause

Place it once in your initial scene.

Why serialize the inspector fields?

To let designers configure:

Autosave interval

Whether autosave is enabled

4. SerializableDictionary

Unity cannot serialize dictionaries.
This class converts a dictionary into two lists (keys and values) for Unity serialization.

It supports:

Add

Remove

Index access

TryGetValue

You use it like a normal dictionary.

🧠 Migration System Explained (VERY IMPORTANT)
Why migrations exist

When you change SaveData between versions:

Add new fields

Rename fields

Remove fields

Change types

Add lists/dictionaries

Old save files will NOT match your new structure.

This causes:

Data loss

Values becoming 0 or null

Missing fields

Fully broken saves

Migrations fix old saves during load.

Migration System Structure
public static class SaveMigrationExample
{
    public static SaveData Migrate(SaveData oldData, int oldVersion, int targetVersion)
    {
        for (int v = oldVersion; v < targetVersion; v++)
        {
            switch (v)
            {
                case 0:
                    // migrate from v0 → v1
                    break;
                case 1:
                    // migrate from v1 → v2
                    break;
                // ...
            }
        }
        oldData.version = targetVersion;
        return oldData;
    }
}

🚀 Practical Examples
✔ 1. Adding a New Field (requires migration)
public int playerLevel = 1; // initializer DOES NOT apply to old saves


Old saves don’t contain this field → Unity sets it to 0.

Migration:

case 1:
    if (migrated.playerLevel == 0)
        migrated.playerLevel = 1;
    break;

✔ 2. Adding a new dictionary (also requires migration)
public SerializableDictionary<string, bool> unlockedHeroes = new();


Old saves set this to null, not new().

Migration:

case 1:
    migrated.unlockedHeroes ??= new SerializableDictionary<string, bool>();
    break;

✔ 3. Renaming a field

Original:

public int coins;


New:

public int gold;


Migration:

case 1:
    migrated.gold = migrated.coins;
    break;


AND remove coins from SaveData after migration is written.

✔ 4. Removing a field

Just delete the field and do nothing.
Unity JSON ignores unknown fields.

🧯 Can I delete old migration cases?
✔ Yes — during development

Since only YOU have save files, you can:

Reset the save system

Delete old migration steps

Reset version back to 1

Delete save.json

This is normal and recommended during early development.

❌ No — after releasing a build to players

Players may have:

Version 1 saves

Version 5 saves

Version 9 saves

If you delete migration steps:

They cannot upgrade

Their save breaks

So after release:
Never delete migrations.

🛠 Development Best Practices

During development:

You can delete the save file whenever needed

You can reset version numbers

You can wipe migrations

You can refactor SaveData freely

After release:

Add migrations only

Never break old saves

Never delete old migrations

📙 Documentation Summary (TL;DR)

SaveData
Defines what gets saved. Version number tracks changes.

SaveManager
Loads, saves, encrypts, backups, and migrates SaveData.

SaveAutoSaver
Ensures loading happens on game start and saves happen regularly.

Migration system
Updates old saves to match new SaveData structure.

During development
You can delete migrations and wipe saves whenever you want.

After release
Keep all migrations forever to avoid breaking player saves.
